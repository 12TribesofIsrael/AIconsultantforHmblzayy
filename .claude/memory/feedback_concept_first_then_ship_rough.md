For any substantive new build deliverable (interactive page, custom tool, multi-segment artifact — not a one-line edit), the order is:

1. **Concept first.** Describe what you'd build — structure, key choices, open questions, where it lives, what the user actually sees — and ask for sign-off before writing code. A 5-line wireframe + a few yes/no questions is usually enough.
2. **Once approved, ship rough.** Build the minimum-viable version that delivers the value, push it live, hand the URL over. Do NOT gold-plate. Polish only if the user requests it after seeing the rough draft land.

**Why:** Thomas corrected this mid-session on May 2 when I jumped straight to building the RV route-planning map without showing him the concept first. His exact words: *"or an exaple what do you think before you build"*. Then on the same deliverable, after agreeing on the concept, he switched gears: *"push it live rough draft real quick while zay is in the chat if he wants more we can polish it up"*. Both halves of the rule come from the same exchange — concept first to align, rough ship fast to seize the moment.

**How to apply:**
- New page / new tool / new automation script / new doc with non-trivial structure → concept first
- Bug fix / typo / one-file edit / mechanical text change / version bump → just do it
- Live-audience window (client actively in chat / news cycle ticking / time-sensitive opportunity) → after concept agreement, prioritize speed of ship over polish. The user will tell you if they want polish.
- Even after sign-off on concept, don't pad the build with "nice-to-haves" the user didn't ask for. The polish round is theirs to request.

**Counter-signal that overrides this rule:** explicit instruction to skip ("just build it", "no need to discuss", "ship now"). Honor that.
